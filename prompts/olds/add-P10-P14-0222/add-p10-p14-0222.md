## 🔍 Step 1 — 미동작 원인 특정

Step 0의 결과를 바탕으로, android 프로젝트에서 入庫処理가 동작하지 않는 원인 레이어를 특정한다.

**체크 순서:**

1. **ViewModel** — `StateFlow` / `LiveData` 가 실제로 데이터를 emit 하는가?
2. **Repository** — 구현체가 있는가, 아니면 interface만 있는가?
3. **API Service** — Retrofit interface 가 정의되어 있고 DI에 등록되어 있는가?
4. **DI 등록** — Hilt / Koin / 수동 DI 어디에도 바인딩이 없는 클래스가 있는가?

```bash
# DI 등록 현황 확인
find sakemaru-handy-android -name "*.kt" | \
  xargs grep -l "@Module\|@Provides\|@Binds\|startKoin\|module {" 2>/dev/null
```

## 🛠 Step 2 — denso 프로젝트에서 로직 이식

### 2-1. 이식 대상 파일 복사

Step 0-5 에서 확인한 denso 의 파일을 android 프로젝트의 대응 경로로 복사한다.

**복사 원칙:**
- `sakemaru-handy-android` 에 **이미 같은 레이어의 파일이 있으면** 덮어쓰지 말고 내용을 비교한 뒤 병합
- `sakemaru-handy-android` 에 **해당 파일이 없으면** denso 파일을 그대로 복사

```bash
# 패키지명 치환 (denso → android 패키지명으로)
# 아래 명령의 패키지명을 실제 값으로 교체 후 실행
find sakemaru-handy-android/app/src/main/java -name "*.kt" \
  -path "*/inbound/*" \
  -exec sed -i 's/com.example.denso/com.example.android/g' {} \;
```

### 2-2. android 프로젝트의 기존 파일과 병합

android 에 이미 있는 파일(특히 ViewModel, Repository)이 있다면:

```
[병합 전략]
- android 파일의 메서드 시그니처 유지
- denso 파일의 내부 구현 로직 사용
- UI에서 observe 하는 StateFlow/LiveData 의 이름이 바뀌지 않도록 주의
```

### 2-3. 누락 의존성 추가

```bash
# denso 의 build.gradle 에서 android 에 없는 의존성 확인
diff <(grep "implementation\|ksp\|kapt" sakemaru-handy-denso/app/build.gradle) \
     <(grep "implementation\|ksp\|kapt" sakemaru-handy-android/app/build.gradle)
```

차이가 있는 항목을 `sakemaru-handy-android/app/build.gradle` 에 추가한다.

## 🔌 Step 3 — DI 연결

### Hilt 사용 시

```kotlin
// sakemaru-handy-android 의 기존 Module 파일에 아래 내용 추가
// (denso 의 Module 을 그대로 복사하면 충돌 가능 — 통합 방식 권장)

@Module
@InstallIn(SingletonComponent::class)
object InboundModule {

    @Provides
    fun provideInboundApiService(retrofit: Retrofit): InboundApiService =
        retrofit.create(InboundApiService::class.java)

    @Provides
    fun provideInboundRepository(
        api: InboundApiService,
        dao: InboundDao          // Room 사용 시
    ): InboundRepository = InboundRepositoryImpl(api, dao)
}
```

### API 설정이 두 프로젝트 간에 다른 경우

```kotlin
// Base URL 이 다를 경우 — Named qualifier 로 분리
@Provides
@Named("inbound")
fun provideInboundRetrofit(): Retrofit =
    Retrofit.Builder()
        .baseUrl(BuildConfig.INBOUND_BASE_URL)
        .build()
```

## 🎨 Step 4 — UI 확인 (수정하지 않음)

`sakemaru-handy-android` 의 UI 파일은 **수정하지 않는다.**  
단, 로직 이식 후 아래 항목만 확인한다:

```
확인 항목 (수정 없이 동작 여부만 체크):
□ ViewModel 의 StateFlow/LiveData 이름이 UI의 collect/observe 와 일치하는가
□ UI 에서 호출하는 ViewModel 메서드명이 이식 후에도 동일하게 존재하는가
□ UI 가 기대하는 데이터 모델(클래스명, 필드명)이 이식된 코드와 일치하는가
```

불일치가 있을 경우 **UI 쪽을 수정하지 말고**, 이식한 로직 쪽의 클래스명·메서드명을 android UI 에 맞게 맞춘다.

## ⚠️ 충돌 발생 시 대응

| 상황 | 대응 |
|------|------|
| android 에 동일 클래스명이 이미 있음 | 내용 비교 후 denso 로직으로 내부 구현만 교체 |
| Room DB 스키마 충돌 | Migration 스크립트 추가 또는 Entity 통합 |
| API BaseUrl / 인증 토큰 방식이 다름 | `@Named` qualifier 로 별도 Retrofit 인스턴스 주입 |
| DI 모듈 중복 | android 기존 Module 에 denso 의 provide 함수만 추가 |
| ViewModel 메서드 시그니처 불일치 | android UI 가 기대하는 시그니처에 맞게 denso 로직을 래핑 |
